## 1. エグゼクティブサマリ

**断言レベル（本レポート内の表記）**

- **仮説**: ベンチマーク観測や経験則からの推測。2週間検証で反証/更新する。
- **要確認**: 代表ヒアリング（`04_insights/representative_interview_one_pager.md`）で確定する前提。
- **案**: 実行プランのたたき台。検証結果で更新する。

### 結論（推奨案｜仮説）

- **推奨**: 戦略類型は **C（ハイブリッド）** を第一候補とする（代表は“翻訳者”、組織は“品質の再現装置”）。
- **理由**: 成功ベンチマークで観測された「思想→制度（仕組み/根拠）」の二段構えが、ターゲット境界（煽り忌避・透明な見立て嗜好）と整合し、失敗ベンチマークが示す地雷（信用危機/表示規制/許認可根）に耐性を作りやすい。

### 意思決定（採る/捨てる/条件付きで採る｜仮説）

詳細は `03_analysis/win_patterns_and_landmines.md` を参照。

- **採る**: 思想→制度の二段構え / 透明な見立て（プロセス可視化） / “伴走”を制度として列挙（属人性を下げる）
- **捨てる（禁止）**: 根拠の弱い「率/社数」断言 / 許認可・守秘・利益相反を発信より後回し / 初動体制なしの強い主張依存
- **条件付きで採る**: 数値訴求 / 強い主張 / 低摩擦入口（匿名相談/棚卸し/診断）
  - 条件: 定義・母数・期間・例外・比較対象が一次で辿れる／公開前レビューを運用する／対応上限と断る基準を先に決める

### Go/No-Go（2週間で揃える材料｜案）

詳細は `04_insights/concept_test_plan.md` を参照。

- **受容性（量）**: 「相談したい」系の具体反応が一定数（例: 3–5件/2週間）
- **受容性（質）**: 相談内容の TagA/B/C が 50%以上（原石層への当たり）
- **工数対効果**: 対応が代表稼働上限内で回る（上限超過・炎上リスク増の兆候がない）
- **安全性**: 数値/実績断言がガードレールに従っている（脚注テンプレ＋公開前レビューが回る）

### 期待効果（短期・中長期）

- **短期（2〜4週間）**: 受容性（相談したい）と、低摩擦入口（匿名相談/棚卸し/診断）の成立を検証し、Go/No-Goの材料を揃える。
- **中長期（3か月〜）**: 指名理由（見立てプロトコル）を資産化し、属人性を抑えつつ集客→成約に接続する。

### 直近30/60/90日の優先アクション（案）

- **30日**
  - 代表ヒアリングで前提（ターゲット境界/稼働上限/露出許容度/NG情報）を確定（`04_insights/representative_interview_one_pager.md`）
  - コンセプトテスト（2週間）を実施し、Go/No-Go判断材料を回収（`04_insights/concept_test_plan.md`）
- **60日**
  - 反応が最も良い訴求に寄せて“型”を固定（投稿テンプレ、チェックリスト、ケーススタディ）
  - 低摩擦入口を運用として成立させる（対応SLA/タグ分類/断る基準/情報取扱い）
- **90日**
  - 成約の独自性が説明できる状態に（事例化、プロトコル、レビュー体制）
  - 属人性を下げる（チーム移管できる工程を増やす）

## 2. 背景と前提

### 事業の現状（勝ち筋仮説）

- 新規のエンジニア採用エージェント事業において、広告費投下での獲得は悪手になりやすい前提で、**「候補者＆求人の見立て力（採用解像度）」を核にしたプル型集客（個人ブランディング）**が“指名”に繋がるかを検証する。

### 個人ブランディングが効く理由（信頼のメカニズム）

- 人材領域は「情報の非対称性」が大きく、意思決定は“信頼できる翻訳者”に依存しやすい。
- ただし、個人を前面に出すほど、信用危機（炎上・疑惑）・表示規制・許認可/コンプラの“根”に弱くなる。

### 制約条件（要確認）

- 代表の稼働上限・露出許容度はヒアリングで確定が必要（`04_insights/representative_interview_one_pager.md`）。
- 数値訴求は根拠が揃うまでは断言しない（景表法リスク）。数値を出す場合は「定義/期間/母数/例外/比較対象」を脚注でセットにし、公開前レビューを運用化する（`04_insights/concept_test_plan.md`）。
- X（旧Twitter）に依存しない（運用/取得の不確実性が残る前提）。
- 許認可/守秘/利益相反は“発信より先”に土台として固定する（提供範囲の明文化、情報取扱い、事故時初動）。

## 4. ポジショニングとブランド要素

### 「誰として」語るか（仮説）

- 代表: 候補者の損失（評価の非対称性）を言語化する **翻訳者**
- 組織: 見立ての品質を再現する **プロトコル/レビュー体制**

### 提供価値（独自性）

- “見立て力”をスローガンではなく、チェックリスト/ケースで提示する（`03_analysis/win_patterns_and_landmines.md`）

### 人間味の設計

- 露出度（オタク性・職歴）はヒアリングで上限を決める（`04_insights/representative_interview_one_pager.md` のQ3）。

## 8. リスクとガードレール

運用チェックリスト（提出物・コピペ用）: `05_outputs/appendix_guardrails_checklist.md`

### 8.1 信用危機（個人前面）の吸収（UUUMケースの含意）

- **原則**: 強い主張ほど「説明責任」と「初動」をセットにする（勢いに依存しない）。
- **最小運用**
  - 事故時の初動（連絡先/記録/公開・非公開の切り分け）を事前に決める
  - 反証可能性の低い断言を避け、「見立て観点（透明性）」へ寄せる

### 8.2 数値/実績表示の統制（DYMケースの含意）

- **禁止（原則アウト）**
  - 「あたかも〜」表示（定義/母数/期間/比較対象なしの断言）
  - 社数/求人数等の断言を、定義/カウントルールなしに行う
  - 例外があるのに全称で断言する（例: “書類選考なし”）
- **必須（数値を出すならセット）**
  - 脚注テンプレ: 定義 / 期間 / 母数 / 例外 / 比較対象
  - 公開前レビュー: 根拠一次が辿れるか、全称断言になっていないか、「あたかも」になっていないか

### 8.3 許認可/守秘/利益相反（永信国際ケースの含意）

- **提供範囲の明文化**: 「職業紹介」「募集情報提供」「採用代行（RPO）」「キャリア相談（助言）」のどれを提供するかを先に固定し、許認可/届出が絡む場合は開始前レビュー（法務）を前提化。
- **情報取扱い（候補者/企業）**: 目的外利用の禁止、第三者提供の条件、保管期間、削除手順を運用として定義。面談内容・職務経歴は機微情報として最小共有。
- **利益相反**: 報酬構造がある場合は立場の開示、推薦基準、断り方をガードレール化。

## 9. KPI設計と検証計画

### 9.1 先行指標（市場の受容性）

- 保存/引用（共感・再訪シグナル）
- DM/問い合わせ数
- 「相談したい」系の具体反応（例: 3–5件/2週間をGo寄りの目安）
- 反応の質（“誰に効いたか”が分かるか）

### 9.2 質の観測（原石度タグ）

`04_insights/concept_test_plan.md` のタグで、相談/DM/コメントを分類する。

- TagA: 評価の非対称
- TagB: 環境要因（負債/体制/事業都合）
- TagC: コミュ/政治
- TagD: 単純年収ハック
- TagE: スキル/習慣課題が主

**Go寄り目安**: TagA/B/C が 50%以上（かつ対応上限内で処理できる）。

### 9.3 事業指標への接続（独自性/工数対効果）

- 商談化、成約、紹介（定性でもよいが“指名理由”が言語化されるか）
- 継続支援の可否（代表稼働上限内で回るか）
- 検証計画: `04_insights/concept_test_plan.md` に準拠（2週間→週次レビュー）

## 10. 90日ロードマップ（案）

- **Week 1-2（設計＋2週間検証を開始）**
  - 前提確定: ターゲット境界/稼働上限/露出許容度/NG情報（`04_insights/representative_interview_one_pager.md`）
  - ガードレール運用を固定: 表示統制・守秘・提供範囲・利益相反・初動（`04_insights/concept_test_plan.md` / `05_outputs/appendix_guardrails_checklist.md`）
  - 実験を回す（P4）: メッセージA/B/C、CTA（匿名相談/棚卸し/診断）、独自性（見立てチェックリスト/ケース）を同一フォーマットで比較（`04_insights/concept_test_plan.md`）
- **Week 3-6（検証→型化）**
  - 反応が最も良い訴求に寄せて、投稿テンプレ・CTA導線・レビュー手順を固定（“勝ち筋の型”を作る）
  - 低摩擦入口を運用として成立させる（対応SLA/断る基準/タグ分類/情報取扱い）
- **Week 7-12（拡張＋属人性低減）**
  - ケース（架空/匿名化）を積み、指名理由（見立てプロトコル）を言語化して資産化
  - チャネル/露出の拡張（コラボ/登壇等）を、ガードレール前提で追加
  - チーム移管できる工程を増やす（レビュー体制・チェックリスト運用を前提に属人性を下げる）
